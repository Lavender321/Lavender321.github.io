OSI七层模型

1. 应用层（Application Layer）

  • 功能：为应用软件提供网络服务
  • 协议/服务：HTTP、HTTPS、FTP、SMTP、POP3、IMAP、Telnet、SSH等

2. 表示层（Presentation Layer）

  • 功能：确保一个系统的应用层发送的信息可以被另一个系统的应用层读取，即进行数据的转换、加密和压缩
  • 协议/服务：JPEG、ASCII、EBCDIC、TIFF、GIF、PICT、Encryption、MPEG、MIDI等

3. 会话层（Session Layer）

  • 功能：建立、管理和终止应用程序之间的会话
  • 协议/服务：NetBIOS、SMB、RPC、NFS等

4. 传输层（Transport Layer）

  • 功能：提供端到端的数据传输服务，确保数据的完整性
  • 协议/服务：TCP（提供可靠的、面向连接的传输）、UDP（提供不可靠的、无连接的传输）

5. 网络层（Network Layer）

  • 功能：决定数据的路径选择和转发，实现不同网络之间的数据传输
  • 协议/服务：IP、ICMP、IGMP、IPSec等

6. 数据链路层（Data Link Layer）

  • 功能：在物理网络之上提供数据传输的方法，处理数据帧的错误检测和重发
  • 协议/服务：Ethernet、802.11（无线局域网协议）、PPP、ATM等

7. 物理层（Physical Layer）

  • 功能：处理通过物理媒介传输的原始比特流
  • 设备/标准：电缆、光纤、RJ45、802.3（Ethernet标准）、V.35等

OSI模型为理解网络通信提供了一个有用的概念框架，虽然现实中的网络协议通常不是严格按照这一模型设计（如Internet主要基于TCP/IP模型），但OSI模型的层次划分依然被广泛用于教学和网络设计之中。

TCP与UDP

区别

TCP（传输控制协议）和UDP（用户数据报协议）是两种最基本的互联网协议，它们都位于OSI模型的传输层，提供了不同类型的数据传输服务。它们之间的主要区别在于TCP提供的是可靠的、面向连接的服务，而UDP提供的是不可靠的、无连接的服务。

  1. 连接

    • TCP：面向连接，进行数据传输前需要建立连接。
    • UDP：无连接，发送数据前不需要建立连接。

  2. 可靠性

    • TCP：可靠传输，通过序号、确认应答、重传机制等确保数据完整性。
    • UDP：不保证可靠传输，上层应用需要自己解决数据的完整性问题。

  3. 数据传输方式

    • TCP：面向字节流，数据以字节流的方式发送。
    • UDP：面向报文，发送的数据保持了报文的边界。

  4. 速度和效率

    • TCP：由于需要建立连接和各种控制机制，相对UDP而言，速度较慢、效率较低。
    • UDP：由于没有建立连接的开销和控制机制，速度快、效率高。

  5. 头部开销

    • TCP：头部最小20字节，因为需要更多的控制信息。
    • UDP：头部固定8字节，开销小。

  6. 用例

    • TCP：适用于需要可靠传输的应用，如Web页面请求、文件传输。
    • UDP：适用于对实时性要求高的应用，如在线视频、实时游戏。

TCP的可靠传输机制

TCP通过以下机制来保证数据的可靠传输：

  1. 序列号和确认应答：TCP为每个包分配一个序列号，接收方收到数据后会发送一个确认应答（ACK）。如果发送方在超时时间内没有收到ACK，它会重新发送数据包。

  2. 数据重传：如果数据包丢失或错误，TCP会重新发送该数据包直到接收方正确接收。

  3. 流量控制：TCP使用滑动窗口机制来控制发送方的发送速率，防止接收方来不及接收数据而导致数据溢出。

  4. 拥塞控制：当网络拥塞时，TCP会减少数据的发送速率，避免造成更严重的网络拥塞。TCP使用几种算法（如慢启动、拥塞避免、快重传、快恢复）来动态调整窗口大小。

  5. 有序数据传输：即使数据包在传输过程中的顺序被打乱，接收方也能按照正确的顺序组装数据。

  6. 错误检测：TCP头部和数据部分都包含了校验和，用于检测数据在传输过程中的任何变化，保证数据的完整性。

通过这些机制，TCP能够提供一种可靠的数据传输服务，在数据传输过程中即使发生了丢包、错误、顺序错乱或者网络拥塞等情况，也能确保数据最终能够完整无误地到达接收方。

HTTP与HTTPS的区别

HTTP（超文本传输协议）和HTTPS（安全超文本传输协议）是互联网中用于数据传输的两个基本协议。它们之间的主要区别在于HTTPS提供了数据加密、身份验证和完整性保护，而HTTP没有。

  1. 安全性：

    • HTTP：不安全，数据以明文形式传输，容易被窃听、篡改。
    • HTTPS：通过SSL/TLS协议加密数据，提供安全保证，难以被窃听和篡改。

  2. 端口：

    • HTTP：通常使用80端口。
    • HTTPS：通常使用443端口。

  3. 性能：

    • HTTP：无需加密，相对更快。
    • HTTPS：加密解密过程增加了计算开销，可能稍慢，但随着技术进步，这种差距越来越小。

  4. 成本：

    • HTTP：无需购买SSL证书。
    • HTTPS：需要购买SSL证书，虽然现在有很多免费的证书提供者（如Let’s Encrypt）。

  5. SEO（搜索引擎优化）：

    • HTTPS：被搜索引擎（如Google）视为排名因素，对SEO更有利。

HTTP常见的状态码

HTTP状态码用于表示服务器对请求的响应状态。以下是一些常见的状态码及其含义：

  • 2xx 成功

    • 200 OK：请求成功。
    • 201 Created：请求成功并且服务器创建了新的资源。
    • 204 No Content：服务器成功处理了请求，但没有返回任何内容。

  • 3xx 重定向

    • 301 Moved Permanently：请求的页面已永久移动到新位置。
    • 302 Found：请求的页面临时移动到新位置。
    • 304 Not Modified：自从上次请求后，请求的页面未修改过。

  • 4xx 客户端错误

    • 400 Bad Request：服务器不理解请求的语法。
    • 401 Unauthorized：请求要求身份验证。
    • 403 Forbidden：服务器拒绝请求。
    • 404 Not Found：服务器找不到请求的网页。
    • 408 Request Timeout：服务器等候请求时发生超时。

  • 5xx 服务器错误

    • 500 Internal Server Error：服务器遇到错误，无法完成请求。
    • 501 Not Implemented：服务器不具备完成请求的功能。例如，它无法识别请求方法时可能会返回此代码。
    • 502 Bad Gateway：服务器作为网关或代理，从上游服务器收到无效响应。
    • 503 Service Unavailable：服务器目前无法使用（由于超载或停机维护）。
    • 504 Gateway Timeout：服务器作为网关或代理，但是没有及时从上游服务器收到请求。

了解这些状态码可以帮助开发者诊断和解决Web应用中的问题。
